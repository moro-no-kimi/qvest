Problem Statement

Students read books when librarians recommend them.
Today that knowledge doesn't scale.

What librarians told us

  • Recommendations rely on personal memory and anecdote
  • District-wide borrowing patterns are invisible to individual librarians
  • Highly effective one-on-one advice doesn't reach the full student body

What the district can provide

  • Checkout history from Destiny Library Manager
  • Full electronic catalog data

Why middle school?

  • Enough borrowing history to support personalization
  • Librarians still strongly influence reading choices
  • Reading identity is still forming — recommendations change behavior
  • Pilot is narrow enough to measure within one semester
The gap is not data. Fulton already captures every checkout event. The gap is a system that turns that borrowing history into personalized, librarian-approved recommendations at scale.

District scale

  • 86,031 students across 102 schools and administrative buildings
  • 19 middle schools — the entire pilot target population in one district
  • 4th largest school system in Georgia; 10,900+ employees
Proposed Solution

A hybrid recommendation platform in four layers

LayerWhat it doesWhy it's needed
1 · Collaborative Filtering Finds books borrowed by students with similar reading histories Primary behavioral signal — proven at scale
2 · Content Similarity Matches on genre, themes, series, and difficulty Covers new titles and students with sparse history
3 · Policy Guardrails Enforces grade-band and district suitability rules Non-negotiable for district approval
4 · LLM Explanations Generates plain-English rationale per recommendation Builds trust with students and librarians
This is an augmentation to Destiny, not a replacement. Destiny remains the system of record for catalog and checkout. This system adds a recommendation layer on top of existing exports.

LLM Design Decision

The LLM explains. It does not rank.

LLM scope

  • Receives only book metadata — title, author, genre, catalog description
  • No student identifiers, no borrowing history in the prompt
  • Produces a one-sentence plain-English recommendation reason
  • Fully local fallback when no API key is set
Example output:
"Because you checked out The Giver and other students who liked thoughtful dystopian stories also borrowed Scythe, you may enjoy this next."

Why this boundary matters

  • Ranking over large bounded inventories is a solved problem — collaborative filtering, not LLMs
  • Student data never leaves district infrastructure
  • Explanation quality can be verified and audited independently of ranking
  • LLM cost is bounded: batch over top-10 candidates per student nightly

Privacy boundary

  • LLM prompt contains zero student identifiers
  • All student data stays within district-controlled infrastructure
  • API calls contain only catalog metadata

Librarian

  • See why each title was suggested (provenance + confidence)
  • Approve · Replace · Pin · Suppress actions
  • Audit trail of all changes with timestamp
  • Status labels: Needs review, Pinned, Auto-published

District Leader

  • Pilot adoption summary
  • Exposure and approval metrics
  • Checkout lift trends by school and grade
  • Data freshness timestamps on every metric
  • Separates directional signals from decision-grade findings

Evaluation structure

  • Baseline: same semester, prior year, same schools
  • Comparison group: non-pilot middle schools with similar checkout rates
  • Staggered rollout in two waves within pilot schools
  • Midpoint evaluation at week 6

Threats to validity — controlled

  • Seasonality → same calendar window prior year
  • Librarian effort confound → log AI-driven vs direct suggestions separately
  • Selection bias → choose typical checkout-rate schools, not top performers
Privacy & Data Governance

Five controls. Student data never leaves district infrastructure.

#ControlWhat it means in practice
1 De-identified student IDs Recommendation pipeline uses randomized surrogate keys. Names resolved only at render time, scoped to librarian's school.
2 LLM prompt boundary Prompts contain only book metadata — title, author, genre, description. No student identifiers or borrowing history.
3 Data locality All computation within district-controlled infrastructure. External LLM API calls contain zero student data.
4 Role-scoped access Librarians see only their school. District dashboard shows aggregate metrics — no student-level drill-down.
5 Minimal auditable retention Raw exports deleted after normalization. Interaction table retained for pilot semester + one comparison period.
Authentication specifics: Students authenticate via ClassLink SSO at launchpad.classlink.com/fcs using ID#@fcstu.org addresses. Microsoft Entra ID is the required identity provider — the district's M365 enterprise agreement is FERPA compliant. No local passwords. No new identity silo.

Rollout Plan

14 – 20 weeks from kickoff to expansion decision

Phase 0 — Discovery2 weeks
Confirm Destiny export fields and privacy constraints. Choose 3–5 pilot middle schools. Define checkout-rate baseline. Validate ClassLink launch flow and Destiny deep-link behavior.
Phase 1 — Pilot Build4 – 6 weeks
Build ingestion pipeline. Implement hybrid engine. Stand up student, librarian, and district surfaces. Validate recommendation quality with librarians. Instrument exposure logging and checkout attribution from day one.
Phase 2 — Pilot Operation8 – 12 weeks
Run recommendations during the semester. Monitor checkout lift weekly. Tune thresholds and explanations. Midpoint evaluation at week 6 — pause if approval rate < 50% or lift < 5%. Collect qualitative feedback.
Phase 3 — Expansion Decision2 weeks
Review pilot performance against primary and secondary metrics. Decide whether to expand to all 19 middle schools. Confirm production hardening priorities.
Rollout is staggered deliberately: two waves within pilot schools to create a natural comparison group and catch quality issues before full exposure.
Risks & Mitigations

Four identified risks. Each has a concrete mitigation in the design.

RiskSeverityMitigation built into the design
Poor recommendation quality with sparse data Medium Content similarity + popularity priors fill cold-start gaps. Middle school chosen specifically for better interaction density than elementary.
Low staff trust in AI recommendations High Librarians stay in the review loop. Provenance and explanation visible for every title. System positioned as assistive — human override always wins and is logged.
Privacy and governance concerns High Five-control privacy design (slide 10). LLM sees zero student data. Role-scoped access. No local passwords or identity silo.
District concern about application sprawl Medium Dr. Phillips reduced application inventory from ~7,850 systems to under 500 at a prior district. This system augments Destiny — it doesn't replace it. Fits ClassLink + Entra ID. Zero new identity surface.
